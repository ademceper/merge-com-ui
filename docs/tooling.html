01 — Dead Code Detection
Kullanılmayan Dosya, Fonksiyon & Export
ESLint tek dosya içine bakar. Bu araçlar projenin tamamına bakar — graph tabanlı analiz.
Knip ✂️
ZORUNLU Monorepo

2026'nın dead code silahı. ts-prune ve depcheck'in üstünde hem archived hem de Knip'i öneriyor. Yaptıkları:

  • Hiçbir yerde import edilmeyen dosyalar
  • Kullananı olmayan TypeScript export'lar
  • package.json'da kurulu ama hiç kullanılmayan npm paketleri
  • package.json'da unlisted (gizli bağımlı) paketler
  • Kullanılmayan class member ve enum member'lar
  • devDependencies ile dependencies karışıklığı

Vercel'de ~300k satır kodu silmek için kullanıldı. TanStack monorepo'ların tamamında aktif.

# Anında başla — zero-config
pnpm add -D knip
# İlk çalıştırma — tüm raporu gör
npx knip
# Otomatik fix — dikkatli kullan!
npx knip --fix
# Sadece unused deps listesi (CI için)
npx knip --issues unused-dependencies
// knip.json — monorepo konfigürasyonu { "workspaces": { "apps/web": { "entry": ["src/app/**/*.tsx"] }, "apps/admin": { "entry": ["src/main.tsx"] }, "packages/*": { "entry": ["src/index.ts"] } }, "ignoreDependencies": ["typescript"], "ignorePatterns": ["**/*.stories.tsx"] }
💡

CI Stratejisi: İlk sprint'te sadece rapor al (fail etme), ignore list'i ayarla. Sonra --reporter compact ile CI'a ekle. Yeni dead code merge edilmesin.

ts-prune
ARCHIVED Knip'e geç

Eski dead code aracı. Artık maintenance mode — yeni özellik almıyor. Monorepo desteği yok. ts-prune reposu kendisi Knip'i öneriyor. Mevcut projede varsa Knip'e migrate et.

# Migration
pnpm remove ts-prune
pnpm add -D knip
⛔

Yeni projede kullanma. ts-prune → depcheck → knip sırasıyla gelişti. 2026'da doğrudan Knip başla.

02 — Circular Dependency Detection
Döngüsel Bağımlılık Tespiti
A→B→C→A döngüleri: Webpack/Vite HMR sorunları, başlatma hataları, spaghetti mimari.
Madge
Görselleştirme CI Friendly

Module bağımlılık grafiği + circular dependency tespiti. SVG/PNG görsel çıktı üretiyor. Bağımlılığın nerede başlayıp nerede döndüğünü gösteriyor.

# Circular deps listele
npx madge --circular --extensions ts,tsx ./src
# SVG görsel üret (Graphviz gerekli)
npx madge src/index.ts --image dep-graph.svg
# Yalnızca circular deps SVG
npx madge --circular --image circular.svg src/
// package.json scripts { "scripts": { "check:circular": "madge --circular --extensions ts,tsx src/", "check:graph": "madge --image dep-graph.png src/index.ts", "check:orphans": "madge --orphans src/index.ts" } }

Çıktı örneği:

✖ Found 2 circular dependencies!
1) store/auth.ts → api/client.ts → store/auth.ts
2) ui/Button.tsx → hooks/useTheme.ts → ui/Button.tsx
📌

TypeScript type import'ları atlayabiliriz: --detective-options '{"ts":{"skipTypeImports":true}}' — sadece runtime circular'ları bul.

dpdm
Hızlı ESM/CJS

Madge'e alternatif, büyük kod tabanlarında çok daha hızlı. CommonJS ve ESM desteği. CLI çıktısı temiz, JSON çıktı mevcut. Graphviz gerektirmez.

npx dpdm --no-warning src/index.ts
# Sadece circular deps
npx dpdm --circular src/index.ts
# JSON çıktı (CI için)
npx dpdm --exit-code circular:1 src/index.ts
vite-plugin-circular-dependency
Vite Build-time

Vite dev server ve build sırasında circular deps'leri otomatik yakalar. Madge'e gerek kalmaz — geliştirme sürecinde terminal'de anında uyarı.

// vite.config.ts import circularDeps from 'vite-plugin-circular-dependency' export default { plugins: [ circularDeps({ circularImportThrowErr: false, // dev'de warn }) ] }
⚠️

Sık circular dep nedeni: packages/ui/index.ts barrel dosyasından re-export + aynı paketteki component o barrel'ı import ediyor. Çözüm: component doğrudan kaynak dosyayı import etsin, barrel'ı değil.

03 — Bundle Size Enforcement
Bundle Bütçesi & PR Kontrolü
Her PR'da bundle büyümesini yakala. "Çalışıyor" demeden önce kaç KB eklendi?
size-limit
ÖNERİLEN CI GitHub PR

Performans bütçesi aracı. Her commit'te gerçek kullanıcı maliyetini hesaplar — sadece dosya boyutu değil, tarayıcı parse + execute süresi (headless Chrome). Limiti aşan PR'ları bloklar.

// package.json { "size-limit": [ { "path": "dist/main.js", "limit": "100 KB" // gzip'd }, { "path": "dist/vendor.js", "limit": "200 KB" }, { "path": "dist/main.js", "limit": "500 ms", // parse+exec süresi "name": "JS Execution Budget" } ], "scripts": { "size": "size-limit", "size:why": "size-limit --why" // neden bu kadar büyük? } }
# .github/workflows/size-limit.yml name: Size Limit on: pull_request jobs: size: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: andresz1/size-limit-action@v1 with: github_token: ${{ secrets.GITHUB_TOKEN }} # PR'a otomatik yorum: "+5.2 KB → ❌ limit aşıldı"
💡

--why flag: npx size-limit --why ile hangi dependency ne kadar yer kaplıyor tam liste görürsün. Yanlış import edilen büyük kütüphaneleri bu şekilde tespit et.

bundlewatch
CI GitHub PR

size-limit'e alternatif. Webpack, Vite, Rollup output'larını izler. PR yorumlarında önceki build ile karşılaştırma.

// .bundlewatch.config.json { "files": [ { "path": ".next/static/**/*.js", "maxSize": "100kB" }, { "path": ".next/static/**/*.css", "maxSize": "20kB" } ] }
@next/bundle-analyzer
Next.js Görsel

Next.js için interaktif treemap. Her chunk'ın içinde ne var, hangi paket ne kadar yer kaplıyor. ANALYZE=true pnpm build ile aktif.

// next.config.ts import bundleAnalyzer from '@next/bundle-analyzer' const withBundleAnalyzer = bundleAnalyzer({ enabled: process.env.ANALYZE === 'true' }) export default withBundleAnalyzer(nextConfig)
rollup-plugin-visualizer
Vite Görsel

Vite ve Rollup için bundle treemap. Sunburst, treemap, flamegraph seçenekleri. Hangi modülün ne kadar yer kapladığını renkli görselleştirme.

// vite.config.ts import { visualizer } from 'rollup-plugin-visualizer' plugins: [ visualizer({ open: true, gzipSize: true }) ]
04 — Package Quality & Audit
Paket Kalite Kontrolü
Bir npm paketi kullanmadan önce: doğru mu yayınlanmış, ESM/CJS sorunu var mı, tipleri doğru mu?
publint
Library'ler için package.json

npm paket yayınlama linter'ı. Kendi packages/ içindeki kütüphaneleri yayınlamadan önce veya üçüncü parti bir paketin sorununu araştırırken kullan. Şunları kontrol eder:

  • ESM mi yoksa CJS mi yazılmış, doğru mu yorumlanıyor?
  • exports field'i tüm bundler'larda düzgün çalışıyor mu?
  • Vite, Webpack, Rollup, Node.js uyumluluğu
  • Yayınlanan dosyalar .npmignore'a düşmemiş mi?
  • Missing veya broken entry points
# Kendi paketini kontrol et
npx publint
# Herhangi bir npm paketini kontrol et
npx publint zod
npx publint @tanstack/react-query
# Online: publint.dev/<paket-adı>
🌐

publint.dev — online versiyon. Bir paketin neden import'ta sorun çıkardığını araştırırken URL'e yaz: publint.dev/react-query

@arethetypeswrong/cli
TypeScript ESM/CJS Types

"Are the types wrong?" — npm paketinin TypeScript tiplerini test eder. ESM/CJS type masquerading, yanlış .d.ts yapılandırması, module resolution sorunlarını bulur. publint ile birlikte kullan — farklı sorunları yakalar.

npx @arethetypeswrong/cli zod
npx @arethetypeswrong/cli @tanstack/react-query
# Kendi paketini test et
npx @arethetypeswrong/cli -P .
💡

Online: arethetypeswrong.github.io — paketin adını yaz, anında rapor. Bir paketin CJS/ESM tip sorunu çıkarıyorsa buradan araştır.

node-modules-inspector
Görsel pnpm

node_modules içini görselleştir. Hangi paket hangi versiyonu, neden bu kadar büyük, hangi paketi kim import ediyor. pnpm'i tarayıcıda simüle ediyor. pnpm dlx node-modules-inspector ile anında çalışır.

pnpm dlx node-modules-inspector
# Tarayıcıda localhost açılır
syncpack
Monorepo Version Drift

Monorepo'da her workspace'de aynı paket farklı versiyonla kurulmuşsa bundle'a iki kez girer (React 18 + React 19 aynı anda). Syncpack tüm package.json dosyalarındaki versiyon tutarsızlıklarını bulur ve düzeltir.

npx syncpack list-mismatches
# Otomatik düzelt
npx syncpack fix-mismatches
05 — Dependency Security
Bağımlılık Güvenliği & Güncelleme Takibi
Hangi paket ne zaman güncellenmeli? Hangi paketin CVE'si var?
pnpm audit
Built-in

pnpm'in yerleşik güvenlik taraması. npm registry'nin CVE veritabanına karşı kontrol. CI'da --audit-level=high ile kritik açıkları yakala.

pnpm audit --audit-level=high
# CI'da fail et
pnpm audit --audit-level=critical
taze / npm-check-updates
Güncelleme

Hangi paketlerin yeni versiyonu var? taze — çok daha temiz UI. ncu — klasik, monorepo desteği. İkisi de update öneriyor ama apply etmeni istiyor.

npx taze
# veya
npx npm-check-updates -i
Renovate Bot
Otomatik PR

Dependency güncellemelerini otomatik PR olarak açar. Dependabot'tan çok daha akıllı — monorepo, lockfile güncelleme, gruplandırma, test geçince auto-merge. GitHub App olarak kurulur.

socket.dev
Supply Chain

npm paket supply chain ataklarını tespit eder. Yeni kurulan bir paketin başka bir güvenli paketi taklit edip etmediğini, typosquatting yapıp yapmadığını kontrol eder. GitHub App.

06 — pnpm v10 Yeni Özellikler
pnpm v10 — Security by Default
2025'te yayınlanan pnpm v10, güvenlik modelini köklü değiştirdi.
Security by Default
Breaking Change v10.0

pnpm v10'da preinstall ve postinstall scriptleri artık default olarak çalışmıyor. Supply chain saldırılarının büyük çoğunluğu bu script'lerle gerçekleşiyordu. İzin vermek istediğin paketleri açıkça belirtmen gerekiyor.

# .npmrc allow-scripts=esbuild,@esbuild/*,sharp,canvas
// package.json — alternatif (allowBuilds) { "pnpm": { "allowBuilds": ["esbuild", "sharp", "@img/sharp-linux-x64"] } }
🚨

Migration Notu: pnpm v9'dan v10'a geçince postinstall script'e ihtiyaç duyan paketler (esbuild, sharp, canvas) build sırasında sessizce başarısız olabilir. pnpm install sonrası warning'leri oku.

Runtime Management
Yeni 2025

pnpm artık Node.js, Deno ve Bun versiyonlarını da yönetiyor. package.json'da devEngines.runtime belirterek takımdaki herkesin aynı runtime versiyonunu kullanması garantileniyor — "Works on my machine" bitti.

// package.json { "devEngines": { "runtime": { "name": "node", "version": "22.x" } } } // pnpm otomatik doğru versiyonu indirir
Config Dependencies
Yeni Monorepo

Dependency graph resolve edilmeden önce yüklenen özel paketler. Shared .pnpmfile.cjs hook'larını, merkezi patch dosyalarını ve allowBuilds listesini repo genelinde paylaşmak için ideal.

08 — Online Araçlar
Tarayıcıda Kullanılan Araçlar
Kurulum gerektirmeyen, link paylaşarak takımla kullanılabilecek araçlar.
  • bundlephobia.com Herhangi bir npm paketinin gzip bundle boyutunu gösterir. Alternatif paket karşılaştırması. "moment.js (70KB) vs dayjs (2.6KB)" gibi seçimler için temel referans.
  • bundlejs.com Import et, online bundle'la, boyutu gör. Tree-shaking'i test et: import { format } from 'date-fns' yazınca sadece o fonksiyonun boyutu çıkar. Badge için de kullanılır.
  • publint.dev Herhangi bir npm paketinin yayın yapılandırmasını analiz et. ESM/CJS sorununu araştırırken ilk gittiğin yer burası olmalı.
  • arethetypeswrong.github.io Paketin TypeScript type dosyalarının doğru yapılandırılıp yapılandırılmadığını kontrol et. ESM/CJS type masquerading sorunları için.
  • pkg-size.dev Paketin npm tarball boyutu, publish edilen dosyalar, tree-shaking sonucu. Bundlephobia'dan daha detaylı dosya bazlı analiz.
  • npm.devtool.tech npm paketlerini compare et. Side-by-side özellik karşılaştırması, bundle size, download trend, GitHub stars.
  • npmgraph.js.org Bir npm paketinin dependency graph'ini görselleştir. "Bu paket gerçekte ne kadar bağımlılık getiriyor?" sorusuna cevap.
  • squoosh.app Google'dan görsel optimizasyon aracı. WebP, AVIF, JPEG XL dönüşümü. next/image veya expo-image'a geçmeden önce statik görselleri optimize et.
  • TypeScript Playground play.typescriptlang.org — TS kodu paylaş, ekiple debug et. Compiler options değiştir, output'u gör.
  • regex101.com Regex test ve debug. Zod pattern'larını, validation regex'lerini test et. Group açıklamaları ve step-by-step eşleşme görsel.

10 — TypeScript Tooling
TypeScript Derleme & Analiz
Type check hızı, strict ayarları, önemli compiler flagler.
tsgo (TypeScript Native Port)
2025 Yeni 10x Hızlı

Microsoft'un TypeScript'i Go'ya port etme projesi. 2025'te preview olarak yayınlandı. tsc'den 10x daha hızlı type checking. Production'da henüz değil ama izle — 2026 sonu stable olabilir.

# Şu an preview
npm install -g @typescript/native-preview
tsgo --version
tsc --build mode
Monorepo

Monorepo'da her package için ayrı tsconfig.json + project references. tsc -b sadece değişen paketleri derler. Turborepo cache ile birleşince type check dramatik hızlanır.

// packages/ui/tsconfig.json { "extends": "../../packages/config/tsconfig.base.json", "compilerOptions": { "composite": true, // project references için "outDir": "dist" }, "references": [ { "path": "../schemas" } // bağımlı paketler ] }
verbatimModuleSyntax
TS 5.0+

import type zorunluluğu. Type-only import'ları import type ile yazmaya zorlar — tree-shaking ve ESM/CJS interop için kritik. tsconfig.base.json'a ekle.

{ "compilerOptions": { "verbatimModuleSyntax": true, "moduleResolution": "bundler", "strict": true, "noUncheckedIndexedAccess": true // arr[0] → T | undefined } }
@total-typescript/ts-reset
Type Safety

TypeScript'in "yanlış" davrandığı yerleri düzeltir. JSON.parse() artık unknown döndürür (any değil), Array.filter(Boolean) doğru tipte, fetch response düzgün typed. Tek satır ekle.

// global.d.ts /// <reference types="@total-typescript/ts-reset" />
11 — Küçük Ama Kritik Detaylar
Gözden Kaçan Her Şey
Tek araç değil ama mutlaka bilmen gereken pratikler ve küçük kütüphaneler.
npm-run-all2
ScriptParallel

Script'leri paralel veya sıralı çalıştır. run-p tsc lint lint:format ile üçü aynı anda. Concurrently'ye göre daha temiz output, log interleaving yok. "2" eki — npm-run-all'ın aktif fork'u.

{ "scripts": { "check": "run-p tsc lint test", "dev:all": "run-p dev:web dev:admin" } }
cross-env
Cross-Platform

Environment variable'ları Windows'ta da çalışacak şekilde set et. NODE_ENV=production Linux/Mac'te çalışır ama Windows'ta çöker. cross-env NODE_ENV=production her yerde çalışır.

dotenv-cli / dotenvx
Env

dotenv-cli: .env.local gibi farklı env dosyalarını script'te belirt. dotenvx: çok dosyalı env yönetimi, şifreleme, .env overriding. dotenvx run --env-file=.env.local -- pnpm dev

why-is-node-running
Debug

Node.js process neden kapanmıyor? Hangi async işlem açık kalıyor? SIGINT'e rağmen kapanmayan test veya dev server'ları debug etmek için. node -r why-is-node-running script.js

depcheck (artık Knip'e geç)
Archived

package.json'daki kullanılmayan dependency'leri bulur. Knip daha kapsamlı, ama küçük projeler için depcheck hâlâ işe yarar. Reposu artık Knip'i aktif olarak öneriyor.

is-esm
Küçük

Bir npm paketinin ESM mi CJS mi olduğunu tek komutla kontrol et. npx is-esm lodash → No. Bundle'a girince tree-shaking olmayacağını önceden bil.

Barrel File Sorunları
Anti-Pattern

index.ts barrel dosyaları (her şeyi re-export eden) monorepo'da çok yaygın ama circular dependency'nin ana nedeni. Next.js App Router'da da performans sorunu. Kural: barrel dosyasından import yerine doğrudan kaynak dosyadan import et.

// ❌ Sorunlu — barrel file import { Button, Input, Modal } from '@ui' // ✅ Doğru — doğrudan kaynak import { Button } from '@ui/button' import { Input } from '@ui/input'
patch-package / pnpm patch
Workaround

Upstream'e PR atmayı bekleyemediğin node_module bug'larını geçici olarak patch'le. pnpm v10'da built-in: pnpm patch <package> → düzenle → pnpm patch-commit

pnpm patch some-broken-package@1.0.0
# Düzenle...
pnpm patch-commit /tmp/patch-dir

Kural: Pre-commit hızlı (format, lint) yapılır. Yavaş şeyler (test, type-check) pre-push'a veya CI'a bırakılır. Commit'i engelleyen uzun hook = developer frustration.
